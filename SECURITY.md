# Security Policy

## Supported Versions

| Version | Supported |
|---------|-----------|
| Latest (`main`) | ✅ |
| Older releases | ❌ |

Only the latest code on `main` receives security fixes. If you are running a self-hosted instance, keep it up to date.

## Reporting a Vulnerability

**Please do not open a public GitHub issue for security vulnerabilities.**

Disclosing a vulnerability publicly before a fix is available puts everyone running SecurityBuddy at risk.

### How to Report

1. Go to the [Security tab](https://github.com/therealmogu/securitybuddy/security/advisories/new) of this repository and open a **private security advisory**.
2. Alternatively, send the details directly to the maintainer via GitHub's private messaging.

Include as much of the following as possible:

- Type of vulnerability (e.g. SSRF, XSS, IDOR, injection)
- The affected endpoint or file
- Step-by-step reproduction instructions
- Potential impact
- Any suggested mitigation

You will receive an acknowledgement within **72 hours**. We aim to release a fix within **14 days** for critical issues and **30 days** for moderate ones. We will keep you updated throughout the process.

### Coordinated Disclosure

We follow a coordinated disclosure model. Once a fix is released we will:

1. Publish a GitHub Security Advisory with full details.
2. Credit you in the advisory (or keep you anonymous if you prefer).

We ask that you do not publish details of the vulnerability until the fix has been released and deployed.

## Scope

The following are **in scope** for security reports:

- Authentication and authorisation bypasses
- Server-Side Request Forgery (SSRF) via scan targets
- SQL injection or other injection vulnerabilities
- Cross-Site Scripting (XSS) or Cross-Site Request Forgery (CSRF) bypasses
- Information disclosure (credentials, private scan data, user accounts)
- Insecure handling of OAuth tokens (Gmail integration)
- Rate limiting bypasses that enable abuse at scale

The following are **out of scope**:

- Vulnerabilities in third-party services used by self-hosted instances (Vercel, Neon, Supabase…)
- Issues that require physical access to the server
- Reports generated by automated scanners without a proof-of-concept
- Rate limiting that only affects a single IP
- Self-XSS or issues that require the attacker to already have admin access
- Findings on demo/staging deployments that are not reproducible on `main`

## Responsible Use Reminder

SecurityBuddy is a **defensive** tool. It is designed to help website owners audit their own infrastructure. Using it to scan systems you do not own or have explicit written permission to test may violate computer fraud laws in your jurisdiction. The maintainers accept no liability for misuse.
